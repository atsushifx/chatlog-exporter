## 1. Overview

### 1.1 Purpose

本仕様は、チャットログ (Markdown 形式) を AI が解析し、問題領域ごとのセグメントに分解・分割して
正規化された個別 Markdown ファイルとして出力するモジュール `chatlog/normalize` の
ふるまいルールを定義する。

### 1.2 Scope

本仕様は `chatlog/normalize` の **ふるまいルール** および **分類セマンティクス** を定義する。

実装の詳細はスコープ外とする。

---

## 2. Design Principles

### 2.1 Classification Philosophy

本モジュールは「ルールベースのトピック分類」ではなく、「AI の文脈理解による問題領域の識別」を採用する。
チャットログには複数の技術領域・作業トピックが混在しうるため、固定ルールでは対応不可能な多様性をカバーするために
Claude CLI を活用した AI 駆動の分割判定を行う。

分割の基本単位は「問題領域 (ドメイン)」であり、単一の会話交換ではなく意味的なトピックのまとまりとする。

### 2.2 Design Assumptions

- 入力ファイルは Markdown 形式であり、フロントマターを持つ場合がある (必須ではない)
- 入力ファイルは UTF-8 でエンコードされている
- AI 処理は Claude CLI (`claude -p`) を経由して行われる
- 出力先ディレクトリは入力ディレクトリと分離されている
- 入力ファイルは読み取り専用として扱われる

### 2.3 External Design Summary

> **Source**: 要件定義書 (requirements.md v1.0) の機能要件および設計決定から導出。

#### Feature Decomposition

| Unit               | Responsibility                                           | REQ Coverage         |
| ------------------ | -------------------------------------------------------- | -------------------- |
| 入力収集           | CLIオプションから対象Markdownファイルを収集する          | REQ-F-001, REQ-F-002 |
| AI分割処理         | チャットログ本文をAIに渡し、問題領域セグメントを識別する | REQ-F-003            |
| 出力ファイル生成   | セグメントごとにサマリー＋元ログ抜粋を含むMDを生成する   | REQ-F-004            |
| フロントマター合成 | 元メタデータを引き継ぎ、AIが生成したフィールドを付加する | REQ-F-005            |
| 元ファイル保全     | 入力ファイルへの書き込み操作を行わない                   | REQ-F-006            |

#### Unit Interaction Map

```text
+----------+     +----------+     +--------------+
|  入力収集 | --> | AI分割処理| --> | 出力ファイル  |
+----------+     +----------+     |    生成       |
                                  +--------------+
                                         |
                                         v
                                  +--------------+
                                  | フロントマター|
                                  |    合成       |
                                  +--------------+
```

元ファイル保全は全ユニットに横断する制約であり、入力収集から出力ファイル生成まで
入力ファイルへの書き込み操作が行われないことを保証する。

#### Data Flow Diagram

```text
[CLIオプション] --> [入力収集] --> [AI分割処理] --> [出力ファイル生成] --> [temp/normalize_logs/]
                        |               |
                        v               v
                  [パス検証]      [セグメント識別]
                                        |
                                        v
                                 [フロントマター合成]
```

### 2.4 Non-Goals

> **Derivation**: 以下の項目はすべて REQUIREMENTS Section "Out of Scope" に由来する。

- 元のチャットログファイルの変更・削除 ← REQ: Out of Scope #1

- Obsidian への直接インポート処理 ← REQ: Out of Scope #3

### 2.7 DD to DR Promotion Criteria

> **Purpose**: DD を正式 DR に昇格すべき場合の判断基準。昇格は **人間の判断** によるものであり、これらの基準は参考情報である。

**DD の昇格を検討すべき状況:**

1. **複数仕様への影響** — 当該決定が複数の仕様またはモジュールに影響する
2. **アーキテクチャ上の重要性** — 当該決定が将来の設計選択を制約する
3. **重要な代替案の存在** — 複数の有力な選択肢が存在していた
4. **ステークホルダーへの可視性が必要** — 外部関係者によるレビューが望まれる

**DD として維持すべき状況:**

- 本仕様のみに閉じた局所的な決定である
- 実質的な代替案が存在しなかった
- 根拠が文脈から自明である

> **Action**: 昇格する場合は `/deckrd dr --add` を使用し、DD Status を `Promoted → DR-xx` に更新すること。

## 3. Behavioral Specification

### 3.1 Input Domain

- 入力タイプ: Markdown ファイル (フロントマターを持つ場合がある)
- 前提条件:
  - ファイルは UTF-8 でエンコードされている
  - ファイルは `temp/chatlog/` 配下に存在する (`--agent`/`--year-month` 指定の場合)
  - または任意のディレクトリに存在する (`--dir` 指定の場合)

### 3.2 Output Semantics

- 出力の意味: 問題領域ごとに分割された独立した Markdown ナレッジファイル
- 出力先: `temp/normalize_logs/<agent>/<year>/<year-month>/<project>/`
- 可能な結果:
  - 正常完了: 1 件以上のセグメントファイルが生成される
  - 部分完了: 一部ファイルが失敗しスキップ、完了時に失敗件数がレポートされる
  - 異常終了: 入力パスが存在しない場合にエラー終了する

## 4. Decision Rules

<!--
Rule ID format: R-NNN (sequential, stable)
Rule IDs are referenced in Traceability and Edge Cases.
-->

評価は以下の順序で実行しなければならない (No reordering is permitted):

| Rule ID | Step | Condition                                         | Outcome                                                                                                                     |
| ------- | ---: | ------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------- |
| R-001   |    1 | `--dir` オプションが指定されている                | 指定パス以下の全 Markdown ファイルを収集対象とする                                                                          |
| R-002   |    2 | `--agent` と `--year-month` が指定されている      | `temp/chatlog/<agent>/<year>/<year-month>/` 以下の全 Markdown ファイルを収集対象とする                                      |
| R-003   |    3 | 収集パスが存在しない                              | エラーを報告してシステムを終了する                                                                                          |
| R-004   |    4 | 収集対象ファイルが 1 件以上ある                   | 各ファイルを AI 分割処理に送る                                                                                              |
| R-005   |    5 | AI がファイル本文から問題領域セグメントを識別した | 1 件以上のセグメントが確定する                                                                                              |
| R-006   |    6 | セグメントが確定した                              | セグメントごとにサマリー＋元ログ抜粋を含む Markdown ファイルを出力先に生成する                                              |
| R-007   |    7 | 出力ファイルが生成された                          | 元ファイルのメタデータ (project, date 等) を引き継ぎ、AI 生成の title / log_id / summary を付加してフロントマターを合成する |
| R-008   |    8 | 個別ファイルの AI 処理が失敗した                  | 当該ファイルをスキップし、他ファイルの処理を継続する                                                                        |
| R-009   |    9 | 全ファイルの処理が完了した                        | 失敗件数を含む処理結果をレポートする                                                                                        |
| R-010   |   10 | 処理の開始から終了まで                            | 入力ファイルへの書き込み操作を行わない                                                                                      |
| R-011   |   11 | 同一入力ファイルに対して複数回実行された          | 既存の出力ファイルが存在する場合はスキップし、スキップ件数をレポートする                                                    |
| R-012   |   12 | AI がセグメントを識別した                         | 1 ファイルあたりのセグメント数は最大 10 件に制限する。超過分は破棄する                                                      |

## 5. Edge Cases

| Input                                          | 分類/結果                                          | REQ              | Rationale                                    |
| ---------------------------------------------- | -------------------------------------------------- | ---------------- | -------------------------------------------- |
| 存在しないパスを `--dir` で指定                | エラー終了                                         | REQ-F-001 AC-002 | 収集対象が存在しない場合は処理続行不可       |
| 単一トピックのみのチャットログ                 | セグメント 1 件のみ生成                            | REQ-F-003 AC-005 | AI が 1 件と判定した場合も正常動作として扱う |
| フロントマターなし入力ファイル                 | 引き継ぎフィールドなし、AI 生成フィールドのみ付加  | REQ-F-005        | フロントマター不在は異常ではなく任意条件     |
| 入力ファイルの AI 処理が失敗                   | 当該ファイルスキップ、他ファイルは継続             | REQ-NF-002       | 個別失敗がバッチ全体に波及しない設計         |
| 同一入力に対する再実行                         | 既存出力ファイルをスキップし、スキップ件数レポート | REQ-NF-004       | 冪等性要件の充足 (R-011)                     |
| 1 ファイルから 10 件超のセグメントが識別された | 最大 10 件のみ処理し、超過分は破棄                 | REQ-F-003        | 出力ファイル数の上限制御 (R-012)             |
| 対象ディレクトリに Markdown ファイルが 0 件    | 処理件数 0 件として完了レポートを出力              | REQ-F-001        | 空ディレクトリは収集失敗ではなく 0 件収集    |
